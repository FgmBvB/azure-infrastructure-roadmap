# Automation Best Practices

> [!NOTE]
> This document is part of the Azure Infrastructure Roadmap while preparing for the **Microsoft AZ-104: Microsoft Azure Administrator** certification.
>
> It summarizes Microsoft's recommended practices for automating Azure administration using Azure CLI, Azure PowerShell, Azure Cloud Shell, Azure Resource Manager (ARM), and Bicep.

---

# Overview

Automation is a fundamental capability for modern Azure administration.

Microsoft recommends treating automation as production code by designing repeatable, secure, idempotent, and version-controlled deployments.

Proper automation reduces human error, improves consistency, accelerates deployments, and simplifies infrastructure management at scale.

---

# Prefer Infrastructure as Code

Always define infrastructure using Infrastructure as Code (IaC) instead of manual portal configuration.

Recommended technologies:

- Bicep (preferred)
- ARM Templates
- Terraform (where appropriate)

Infrastructure should always be reproducible.

---

# Prefer Bicep for New Deployments

Microsoft recommends Bicep instead of authoring ARM Templates directly.

Benefits include:

- Cleaner syntax
- Strong typing
- Native modules
- IntelliSense
- Easier maintenance

ARM Templates remain important because they are the deployment format generated by Bicep.

---

# Keep Deployments Idempotent

Automation should produce the same result regardless of how many times it is executed.

An idempotent deployment:

- Creates missing resources.
- Updates existing resources.
- Leaves compliant resources unchanged.

Avoid scripts that require manual cleanup before execution.

---

# Separate Code from Configuration

Keep infrastructure definitions separate from environment-specific values.

Example:

```text
main.bicep

dev.parameters.json

test.parameters.json

prod.parameters.json
```

This improves reuse and simplifies multi-environment deployments.

---

# Store Secrets Securely

Never store:

- Passwords
- Client secrets
- Storage keys
- SAS tokens
- Connection strings

Instead use:

- Azure Key Vault
- Managed Identities
- Secure pipeline variables

Protecting secrets is a core Azure security principle.

---

# Validate Before Deployment

Always validate deployments before execution.

Validation should include:

- Syntax
- Parameters
- Resource types
- API versions
- Permissions

Validation identifies deployment issues before infrastructure changes occur.

---

# Use What-If Analysis

Run a **What-If** operation before deploying production changes.

What-If predicts:

- Resources to be created
- Resources to be modified
- Resources with no changes
- Resources that may be deleted

This significantly reduces deployment risk.

---

# Design Modular Deployments

Divide large deployments into reusable modules.

Typical modules include:

- Networking
- Storage
- Compute
- Security
- Monitoring

Smaller modules are easier to:

- Maintain
- Test
- Reuse
- Troubleshoot

---

# Use Source Control

Treat automation as software.

Store:

- Bicep files
- ARM Templates
- PowerShell scripts
- Azure CLI scripts
- Parameter files

inside Git repositories.

Benefits include:

- Version history
- Collaboration
- Peer review
- Rollback
- Auditability

---

# Use Secure Authentication

Avoid interactive credentials in automation.

Preferred authentication methods include:

- Managed Identities
- Service Principals
- Federated identities
- Azure Cloud Shell (interactive administration)

---

# Apply Least Privilege

Automation identities should receive only the permissions they require.

Avoid assigning:

- Owner

when more restrictive roles are sufficient.

Examples include:

- Contributor
- Virtual Machine Contributor
- Storage Account Contributor
- Reader

---

# Use Consistent Naming Standards

Apply predictable naming conventions across all deployments.

Names should identify:

- Environment
- Workload
- Region
- Resource type

Consistent naming simplifies administration.

---

# Apply Governance Tags

Deploy tags automatically.

Typical tags include:

- Environment
- Owner
- CostCenter
- BusinessUnit
- Application

Tags improve governance and Cost Management.

---

# Use Current API Versions

Always use supported resource API versions.

Benefits include:

- New Azure capabilities
- Security improvements
- Bug fixes
- Better compatibility

Avoid deprecated API versions whenever possible.

---

# Handle Errors Properly

Automation should detect and handle failures gracefully.

Examples include:

- Azure CLI error checking
- PowerShell try/catch
- Deployment validation
- Logging

Never ignore deployment errors.

---

# Log Automation Activities

Maintain logs for:

- Deployments
- Resource creation
- Configuration changes
- Errors
- Execution time

Logs simplify troubleshooting and auditing.

---

# Test Before Production

Validate automation in:

- Development
- Test
- Staging

before deploying to Production.

Production should never be the first execution of new automation.

---

# Review Automation Regularly

Azure evolves continuously.

Review automation to:

- Remove deprecated features
- Adopt new capabilities
- Improve performance
- Simplify code
- Maintain security

Automation should evolve together with the Azure platform.

---

# Enterprise Scenario

A global organization deploys Azure infrastructure across multiple regions.

Administrators:

- Develop reusable Bicep modules.
- Store templates in Git.
- Separate parameter files by environment.
- Retrieve secrets from Azure Key Vault.
- Validate deployments.
- Run What-If before production releases.
- Authenticate using Managed Identities.
- Apply governance tags automatically.
- Review deployment history after every release.

This approach delivers secure, repeatable, and enterprise-ready Infrastructure as Code.

# Best Practices Checklist

- Prefer Infrastructure as Code.
- Use Bicep for new deployments.
- Keep deployments idempotent.
- Separate code from configuration.
- Protect secrets with Azure Key Vault.
- Validate deployments.
- Run What-If before production changes.
- Design modular templates.
- Store automation in source control.
- Use secure authentication.
- Apply least privilege.
- Follow naming conventions.
- Apply governance tags.
- Use current API versions.
- Handle errors properly.
- Log deployments.
- Test before production.
- Review automation regularly.

---

# Common Pitfalls

- Manual portal configuration instead of IaC.
- Hardcoding secrets.
- Skipping deployment validation.
- Ignoring What-If analysis.
- Using Complete deployment mode without impact analysis.
- Assigning excessive RBAC permissions.
- Mixing environment-specific values with infrastructure code.
- Not using source control.
- Deploying directly to Production without testing.

# Key Takeaways

- Automation is a core Azure administration skill and should be implemented using Infrastructure as Code rather than manual portal operations.
- Microsoft recommends Bicep as the preferred authoring language, while Azure Resource Manager remains the deployment engine behind every Infrastructure as Code deployment.
- Secure automation depends on proper authentication, Azure Key Vault integration, least-privilege access, validation, and What-If analysis before production deployments.
- Modular templates, parameter separation, version control, and idempotent deployments improve maintainability, consistency, and scalability.
- Mastering Azure automation practices is essential for enterprise Azure administration and a key objective of the AZ-104 certification.
